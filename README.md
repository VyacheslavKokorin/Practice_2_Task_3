# Practice_2_Task_3
## Требования
Наличие установленного Docker.  
Наличие установленного git для клонирования репозитория.

## Установка и запуск
Перейдите в папку, где будет размещён проект, в терминале. Например:
```
cd ~/Documents
```

Склонируйте репозиторий себе на компьютер
```
git clone https://github.com/VyacheslavKokorin/Practice_2_Task_3.git
```

Перейдите в папку проекта
```
cd Practice_2_Task_3
```

Создайте файл с переменными окружения
```
cp .env_example .env
```
В боевом применении пароли необходимо ОБЯЗАТЕЛЬНО поменять на что-то сложное и уникальное, но в учебном можно оставить так

Запустите контейнеры Docker
```
docker compose up -d
```

После запуска приложение будет доступно по адресу:
```
http://localhost:3000
```

## Анализ выполненной задачи

1. Для проекта был выбран Node.js, Express и MySQL. Приложение и база данных запускаются через Docker Compose.

2. В базе данных созданы связанные таблицы пользователей, постов, подписок, комментариев и тегов. Внешние ключи помогают сохранять целостность данных.

3. В приложении реализованы регистрация, вход и выход. Пароли перед сохранением хешируются с помощью bcrypt, а состояние входа хранится в сессии.

4. Авторизованный пользователь может создавать, редактировать и удалять свои посты. Изменение чужих публикаций запрещено проверкой владельца.

5. Публичные посты отображаются на главной странице и в профиле автора. Для скрытых постов создаётся секретная ссылка с токеном, и в общие списки такие записи не попадают.

6. Требование о посте «только по запросу» можно понять по-разному. Его можно было реализовать через кнопку запроса доступа, но тогда скрытые посты пришлось бы показывать всем без содержимого, что выглядит сомнительно с точки зрения приватности. Поэтому был выбран доступ по ссылке, которую автор сам передаёт нужному человеку.

7. Пользователь может подписываться на других авторов и открывать отдельную ленту их публичных постов. Повторная подписка и подписка на самого себя запрещены.

8. К постам можно добавлять теги. На главной странице поддерживаются фильтрация по тегу и сортировка публикаций по дате.

9. Авторизованные пользователи могут оставлять комментарии к публичным постам. Комментарии выводятся вместе с именем автора и датой создания.

10. Для страниц используется общий HTML-шаблон и CSS-файл. Благодаря этому навигация и оформление не повторяются в каждом маршруте.

## Выявленные ошибки и рекомендации
1. Пользовательский текст вставляется в HTML без экранирования. Перед выводом постов, комментариев, имён и тегов нужно удалять или экранировать опасную разметку.

2. При редактировании поста нельзя изменить его теги. Стоит добавить поле тегов в форму редактирования и сохранять выбранный тег при смене порядка сортировки.

3. На скрытом посте может отображаться форма комментария, хотя обработчик принимает комментарии только для публичных постов. Форму нужно скрывать или разрешить отправку после проверки токена.

4. Создание поста и добавление тегов выполняются без транзакции. Эти запросы лучше объединить, чтобы при ошибке не сохранялись неполные данные.

5. Проверка введённых данных сделана только на базовом уровне. Нужно добавить ограничения длины и формата для имени, email, пароля, постов, тегов и комментариев.

6. Имя базы задаётся и в `.env`, и напрямую в `init.sql`. Лучше использовать одно значение, чтобы изменение настроек не приводило к подключению к пустой базе.

7. В проекте нет автоматических тестов. Стоит проверить регистрацию, права владельца, подписки, скрытые посты, теги и комментарии.

8. Для учебного проекта сессии в памяти подходят. При публикации приложения лучше использовать постоянное хранилище сессий и безопасные параметры cookie.
